## 1. 目标与约束

### 1.1 目标

在不破坏现有 external pool 契约的前提下，补齐内部号池管理能力。

### 1.2 已知现状

1. `accounts.pool_status` 已承载池状态
2. `services/pool.py` + `repositories/pool.py` 已承载外部领取状态机
3. `overview` 已能统计池状态与最近操作
4. `GET /api/accounts` 当前不会把 `pool_status` 直接返回给前端账号列表
5. 现有账号管理接口没有“内部号池维护动作”语义

### 1.3 设计约束

1. 不复用 external pool 接口做内部管理
2. `claimed` 状态必须走单独受控动作
3. 本期优先复用现有 `accounts` / `pool` / `overview` 数据，不新增复杂表结构
4. 前后端集成方式优先对齐当前 `page-dashboard + feature 模块 + Session JSON API` 模式

## 2. 分层落点建议

### 2.1 Route 层

建议新增独立路由模块，例如：

- `outlook_web/routes/pool_admin.py`

职责：

1. 注册内部管理 API
2. 仅服务登录态管理员

### 2.2 Controller 层

建议新增：

- `outlook_web/controllers/pool_admin.py`

职责：

1. 解析筛选参数
2. 校验动作类型与入参
3. 调用 service
4. 返回统一 JSON 响应

### 2.3 Service 层

建议新增：

- `outlook_web/services/pool_admin.py`

职责：

1. 承载内部管理动作规则
2. 定义允许状态集合
3. 对 `claimed` 做保护
4. 需要强制释放时，复用或包装现有 `pool` service 能力

### 2.4 Repository 层

建议新增：

- `outlook_web/repositories/pool_admin.py`

职责：

1. 管理列表查询
2. 单账号状态更新
3. 批量安全更新
4. 最近流转摘要查询

## 3. 接口建议

### 3.1 管理列表查询

建议新增：

- `GET /api/pool-admin/accounts`

建议参数：

1. `in_pool=true|false|all`
2. `pool_status=`
3. `provider=`
4. `group_id=`
5. `search=`
6. `page=` / `page_size=`

建议返回字段：

1. 账号基础信息
2. `pool_status`
3. `claimed_by`
4. `claimed_at`
5. `lease_expires_at`
6. `last_result`
7. `last_result_detail`

### 3.2 单账号动作

建议新增：

- `POST /api/pool-admin/accounts/<id>/action`

建议动作值：

1. `move_into_pool`
2. `move_out_of_pool`
3. `restore_available`
4. `freeze`
5. `retire`
6. `force_release`（增强项，可按 Phase 4 落地）

### 3.3 批量动作（可选，非 MVP）

建议新增：

- `POST /api/pool-admin/accounts/batch-action`

第一版可只支持：

1. `move_into_pool`
2. `move_out_of_pool`
3. `restore_available`
4. `freeze`
5. `retire`

默认不支持：

1. `force_release` 批量化
2. 任意状态强改

## 4. 查询复用与改造建议

### 4.1 是否直接扩展 `GET /api/accounts`

不推荐直接把号池管理语义堆进 `GET /api/accounts`：

1. 当前接口定位是普通账号列表
2. 现有参数以分组、标签、搜索、排序为主
3. 强行加入大量池管理参数会让职责变混

### 4.2 查询复用点

可复用现有 `accounts` repository 的通用查询思路：

1. 搜索条件拼接
2. 分页结构
3. tags/group hydration 逻辑

但号池管理列表需要补出：

1. `pool_status`
2. `claimed_by / claimed_at / lease_expires_at`
3. `last_result / last_result_detail`

### 4.3 最近流转信息

增强项可从：

- `account_claim_logs`

读取最近若干条操作，供详情区展示。

## 5. 动作规则落点

### 5.1 普通状态维护

这类动作建议走 `pool_admin_service` 自己的规则校验后再落库：

1. `NULL -> available`
2. `available -> NULL`
3. `cooldown / used / frozen / retired -> available`
4. `available / cooldown / used -> frozen`
5. `available / cooldown / used / frozen -> retired`

### 5.2 `claimed` 强制释放

`force_release` 不建议直接手写更新 SQL 覆盖，而应尽量复用现有 pool 释放语义：

1. 先读取当前 `claim_token` / `claimed_by`
2. 由内部受控 service 执行释放
3. 写审计与说明字段

如果现有 `release_claim()` 强依赖 `caller_id + task_id` 完整匹配，则本期可能需要新增一条仅内部管理员可用的受控释放路径。

#### 实际落地说明（2026-05-18）

当前实现采用 `pool_admin` 独立受控更新路径完成 `force_release`：

1. 仅允许 `claimed` 状态执行
2. 执行后置为 `available` 并清空 claim 上下文
3. 未直接复用 external pool release 接口的 `caller_id + task_id` 匹配语义

该实现符合“独立受控动作”边界，但仍建议后续评估是否需要进一步复用/对齐底层 release 语义。

## 6. 前端落点建议

### 6.1 页面层

建议新增独立前端模块，例如：

- `static/js/features/pool_admin.js`

推荐模式应直接参考当前 `static/js/features/overview.js`：

1. 由 `navigate('pool-admin')` 进入页面
2. 页面初始化函数独立（如 `initPoolAdmin()`）
3. 模块内维护自己的轻量缓存、筛选状态和刷新逻辑

### 6.2 模板层

建议在：

- `templates/index.html`

中增加独立页面容器，而不是把复杂管理区塞进概览 Tab 内容区。

建议形态：

1. 新增 `id="page-pool-admin"` 的页面容器
2. 结构风格参考当前 `page-dashboard` 的大页组织方式
3. 不新增新的独立 HTML 模板页面

### 6.3 交互复用点

可复用：

1. 当前账号分页样式
2. 当前 toast / modal 反馈体系
3. 当前表格 / 卡片 UI 组件风格
4. 当前 `fetch -> JSON -> 局部刷新` 的内部管理接口调用模式

### 6.4 后端接口风格对齐

建议内部号池管理接口直接参考当前 overview / account 管理接口风格：

1. 继续使用 `@login_required`
2. Controller 直接返回 JSON
3. Repository 负责查询与落库
4. Service 负责动作规则和状态保护
5. 不额外引入 SSE / WebSocket / 长轮询

## 7. 风险点

### 7.1 语义风险

最大风险不是 SQL，而是误把任务语义做成管理员按钮：

1. 强制完成
2. 手工过期回收
3. 任意状态互改

### 7.2 并发风险

如果管理员在外部任务执行中修改 `claimed` 账号，容易造成状态歧义。

因此第一版必须：

1. 默认保护 `claimed`
2. 把人工介入收敛到单独动作

### 7.3 范围风险

若第一版同时上批量、详情、审计、复杂筛选，容易把文档需求一次拉太大。

建议严格按：

1. MVP
2. 增强项
3. 后续扩展

三层推进。

---

## 8. 推荐实施顺序

1. 先补内部查询接口
2. 再补单账号动作接口
3. 再接前端管理页 MVP
4. 再补强制释放与审计
5. 最后再评估批量动作
