## Phase 1: 后端重试与 429 退避

**目标**：为 `test_refresh_token_with_rotation()` 增加重试能力，遇到限流时自动退避，避免单次请求失败就导致整个批量刷新停滞。

**涉及文件**：`outlook_web/services/graph.py`

### Task 1.1：增加 `max_retries` 参数
- [ ] 为 `test_refresh_token_with_rotation()` 新增 `max_retries: int = 3` 参数
- [ ] 保持向后兼容：默认行为不变，不显式传入时启用重试

### Task 1.2：实现指数退避重试逻辑
- [ ] 外层包裹 `for attempt in range(max_retries + 1):` 循环
- [ ] 异常时（`requests.exceptions.RequestException`）等待 `2^attempt` 秒后重试
- [ ] 最后一次尝试仍失败则返回错误

### Task 1.3：实现 429 限流处理
- [ ] 当 `res.status_code == 429` 时：
  - [ ] 读取 `Retry-After` 响应头并转换为整数
  - [ ] 若存在则按该值等待；若不存在则按指数退避等待
  - [ ] 在 `attempt < max_retries` 时执行重试
- [ ] 非 429 的 4xx 错误（400/401/403）不再重试，直接返回失败

### Task 1.4：缩短单次超时
- [ ] 将 `timeout=30` 调整为 `timeout=15`（批量场景下更合理）
- [ ] 评估是否需区分连接超时与读取超时（如 `timeout=(5, 15)`）

### Task 1.5：优化错误信息
- [ ] 重试过程中记录每次失败原因（用于最终返回的 error_msg）
- [ ] 最终错误信息包含重试次数摘要（如"请求异常（已重试3次）：..."）

### Task 1.6：运行测试
- [ ] `python -m compileall -q outlook_web/services/graph.py`
- [ ] `python -m unittest tests.test_refresh_outlook_only -v`

## 风险与注意事项

| 风险点 | 影响 | 缓解措施 |
|-------|------|---------|
| 重试逻辑引入额外延迟 | 批量刷新总时长增加 | 控制 max_retries=3，退避时间可控 |
| timeout 缩短导致弱网环境失败率上升 | 代理/慢速网络下更多失败 | 评估是否区分连接超时与读取超时 |
| 前端 AbortController 兼容性 | 旧版浏览器不支持 | 项目目标浏览器为现代浏览器，兼容性好 |
| delay 前移至循环开头增加总时长 | 50 个账号多增加 ~250s | 可保持循环末尾 delay，仅在请求前加 jitter |
| 修改公共文件引入回归 | 影响其他刷新路径 | 每个 Phase 后运行定向 + 全量测试 |
