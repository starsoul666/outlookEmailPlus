## 1. 功能定义

### 1.1 核心功能

在现有邮箱池能力之上，新增一个面向管理员的**号池管理入口**，用于解决导入后无法持续维护的问题。

第一版按 PRD 的两层口径落地：

1. **最低必须**：
   - 看得见池内 / 池外
   - 能移入号池
   - 能移出号池
   - 能阻止误改 `claimed` 占用中账号
2. **建议增强**：
   - 强制释放
   - 管理动作审计
   - 最近流转信息展示

### 1.2 页面职责边界

本功能不是概览看板增强版，而是独立的管理界面：

1. **概览页**继续负责统计、趋势、最近操作
2. **号池管理页**负责筛选、查看、操作、维护
3. 两者可互相跳转，但不混用职责

### 1.2.1 与当前前端风格对齐原则

第一版应尽量贴合当前项目已有的前端组织方式：

1. 继续使用 `templates/index.html` 中的 `.page` 页面容器模式
2. 继续通过 `navigate(page)` 进行页面切换，而不是额外开一套独立前端入口
3. 前端脚本继续采用 `static/js/features/*.js` 的 feature 模块风格
4. 视觉风格优先复用当前账号页、概览页、toast、modal、分页和按钮样式

### 1.3 第一版最低必须包含

- ✅ 新增号池管理入口
- ✅ 展示池内 / 池外账号列表
- ✅ 单账号移入号池 / 移出号池
- ✅ 展示当前池状态、最近结果、占用信息
- ✅ 对 `available / cooldown / used / frozen / retired / NULL` 做受控维护
- ✅ 对 `claimed` 状态做保护

### 1.4 第一版建议增强项

- ✅ 提供“强制释放”作为增强项
- ✅ 写入审计日志作为增强项
- ✅ 最近流转信息展示作为增强项

### 1.5 本期不包含

- ❌ 强制完成 / 强制成功 / 强制失败
- ❌ 把“过期回收”做成人工按钮
- ❌ 临时邮箱与长期邮箱完全统一维护规则
- ❌ 复杂批量策略与自动编排

## 2. 页面结构设计

### 2.1 页面入口

建议新增一个独立入口，名称为：**号池管理**。

入口要求：

1. 与“数据概览”分离
2. 可从邮箱池概览 Tab 跳转进入
3. 不隐藏在普通账号编辑弹窗深处
4. 入口形态应与当前侧边导航 / 页面切换机制保持一致

### 2.2 页面区域

第一版页面建议由 4 个区域组成：

1. **状态摘要条**
   - 池内总数
   - 可用数
   - 占用中数
   - 冷却数
   - 冻结 / 退休数
2. **筛选栏**
   - 池内 / 池外
   - pool_status
   - provider
   - group
   - email / domain / remark 搜索
3. **账号列表区**
   - 当前页账号明细
   - 行内状态标签与可执行动作
4. **详情 / 最近流转区**（增强项）
   - 展示当前账号最近领取、释放、完成、过期记录

页面交互上，应保持当前系统已存在的“列表 + 分页 + 按钮动作 + toast 反馈”节奏，不额外引入陌生交互范式。

### 2.3 列表字段

第一版列表至少展示：

1. 邮箱地址
2. 分组
3. provider / account_type
4. 普通状态（如 `active / inactive`）
5. 池状态（`NULL / available / claimed / cooldown / used / frozen / retired`）
6. 最近结果（如 `success / verification_timeout / lease_expired`）
7. 当前占用信息（仅 `claimed` 时显示）
8. 最近领取时间
9. 操作列

## 3. 状态与动作设计

### 3.1 动作总表

| 动作 | 适用状态 | 结果 | 说明 |
|---|---|---|---|
| 移入号池 | `NULL` | `available` | 池外账号加入池 |
| 移出号池 | `available / cooldown / used / frozen / retired` | `NULL` | 从池中移除 |
| 恢复可用 | `cooldown / used / frozen / retired` | `available` | 管理员覆盖恢复 |
| 冻结 | `available / cooldown / used` | `frozen` | 暂停参与调度 |
| 退休 | `available / cooldown / used / frozen` | `retired` | 明确下线 |
| 强制释放 | `claimed` | `available` | 增强项，受控动作 |

### 3.2 `claimed` 状态保护

`claimed` 是外部任务占用态，第一版必须单独保护：

1. 不允许通过通用状态下拉直接改动
2. 不允许“移出号池”这类普通动作直接覆盖
3. 若要人工介入，只能进入单独的“强制释放”动作
4. 强制释放应带二次确认

### 3.3 “过期回收”口径

本 FD 明确：

1. “过期回收”是系统自动生命周期，不是管理员主动作
2. 页面可展示该状态流转结果
3. 管理员可对其后置状态执行“恢复可用”等动作
4. 不提供名为“过期回收”的手动按钮

### 3.4 “完成”口径

本 FD 明确：

1. “完成”属于任务结果回写语义
2. 第一版不暴露“强制完成”按钮
3. `used` 这类状态若需重新入池，应通过“恢复可用”表达管理员意图

## 4. 交互规则

### 4.1 单账号操作

管理员在列表中对单个账号执行动作时：

1. 点击动作按钮
2. 弹出确认文案（说明将发生的状态变化）
3. 执行成功后刷新该行状态
4. 给出成功 / 失败反馈

### 4.2 批量操作

批量操作属于增强项，不是第一版最低必须。

若第一版纳入，则要求：

1. 仅对同类可操作状态开放
2. `claimed` 默认不参与批量动作
3. 部分成功时给出逐类反馈

### 4.3 搜索与筛选

用户切换筛选条件后：

1. 列表按服务端结果刷新
2. 分页信息同步更新
3. 切换池内 / 池外时，状态筛选项也应保持一致口径

## 5. 接口语义设计

### 5.1 查询接口

建议新增内部管理查询接口，面向 Session 登录管理员，不复用 external pool 接口。

建议能力：

1. 查询池内 / 池外账号
2. 按状态、provider、group、关键词分页过滤
3. 返回基础池状态字段与最近流转摘要

返回风格应尽量贴近当前内部管理接口：

1. 使用 JSON 响应
2. 与现有分页接口保持一致的分页结构
3. 失败时沿用当前统一错误反馈思路

### 5.2 管理动作接口

建议新增内部管理动作接口，面向 Session 登录管理员：

1. 单账号动作：移入 / 移出 / 恢复可用 / 冻结 / 退休 / 强制释放
2. 批量动作：仅覆盖安全动作
3. 每个动作都应有明确的允许状态集合

### 5.3 审计

管理动作执行后应写入审计，至少包含：

1. 操作者
2. 动作名称
3. 目标账号
4. 变更前状态
5. 变更后状态

## 6. 与现有模块的关系

### 6.1 与账号列表页关系

1. 账号列表页仍负责邮箱日常查看与普通编辑
2. 号池管理页负责池语义维护
3. 普通账号页不再承担完整号池运营职责

### 6.2 与概览页关系

1. 概览页继续展示邮箱池统计
2. 统计卡片可增加“进入号池管理”跳转
3. 概览页不承担重操作

### 6.3 与 external pool 关系

1. external pool 继续面向外部调用方
2. 号池管理页是内部运营入口
3. 两者共享同一套底层状态机，但不是同一类接口

---

## 7. 验收标准

### 7.1 MVP 验收

满足以下四项即可认为主诉求闭环：

1. 管理员能区分池内与池外账号
2. 管理员能把账号移入号池
3. 管理员能把账号移出号池
4. `claimed` 账号不会被误改

### 7.2 第一版完整验收

1. 可对安全状态执行基础维护
2. 能展示当前状态与最近结果
3. 强制释放可作为增强项受控使用
4. 管理动作可追溯

---

## 8. 推荐实施顺序

1. 先做 **MVP**：池内/池外列表 + 移入/移出 + `claimed` 保护
2. 再做 **增强项**：恢复可用 / 冻结 / 退休 / 强制释放 / 审计
3. 最后再看是否需要批量操作与更强详情面板
