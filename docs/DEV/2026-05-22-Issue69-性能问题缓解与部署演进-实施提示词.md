## 当前共识（必须遵守）

### 第一优先级：先做 Phase 1（首屏降载）

你的第一目标不是 gevent，也不是 scheduler 拆分，更不是直接上多 worker。

**当前最优先主线只有一个：**

1. `/api/system/version-check` 改懒加载
2. `/api/groups` 去 N+1
3. `/api/settings` 首屏瘦身
4. 首页请求做“必须 / 可延迟”分层
5. 保持 overview 首屏只拉 summary

### 第二优先级：再评估 gevent

只有在第一优先级完成后，才考虑：

1. 单 worker + gevent 是否值得切
2. 切换后的 SSE / 长连接 / 调度线程风险

### 长期方向：scheduler 拆分后再谈多 worker

在 scheduler 没拆出去之前：

- 不要把“多 worker”当成默认立即可做的优化；
- 不要把问题错误归因成“session 不同步”；
- 要明确 scheduler 重复启动才是当前主要结构性约束之一。

## 推荐实施顺序（严格按顺序）

### Step 1：补齐上下文并确认范围

你需要先输出一段简短理解，确认：

1. 当前问题的真实主线是什么；
2. 为什么不能直接多 worker；
3. 为什么优先做首屏降载。

如果你没有先明确这三点，就不要开始改代码。

### Step 2：落首屏降载最小改造方案

你应优先产出或实现以下改动：

#### 2.1 `version-check` 懒加载

目标：

- 不在首页 `DOMContentLoaded` 时立即触发外网请求；
- 保留用户可见更新提示能力；
- 页面主体先可用，版本提示后补。

重点文件：

- `static/js/main.js`
- `outlook_web/controllers/system.py`

#### 2.2 `GET /api/groups` 去 N+1

目标：

- 把 “load groups + for each group count accounts” 改成聚合式查询；
- 保持原接口结构不变；
- 避免因修复性能问题破坏前端。

重点文件：

- `outlook_web/controllers/groups.py`
- `outlook_web/repositories/groups.py`

#### 2.3 `GET /api/settings` 首屏瘦身

目标：

- 识别首页初始化真正依赖的字段；
- 避免在首页阶段拉取/处理过多配置、解密和统计信息；
- 视情况选择：
  - 缩减现有 `GET /api/settings`；或
  - 新增更轻量的 bootstrap/settings-summary 接口。

重点文件：

- `outlook_web/controllers/settings.py`
- `static/js/main.js`

#### 2.4 首页请求分层

目标：

- 区分首页“必须先返回”的请求与“可延迟补充”的请求；
- 避免重请求和轻请求一起争抢唯一 sync worker。

重点文件：

- `static/js/main.js`
- `static/js/features/overview.js`
- `static/js/features/groups.js`

### Step 3：验证不回退

至少验证：

1. 首页仍能正常打开
2. 默认分组和账号列表仍能正常加载
3. 概览 summary 正常显示
4. 版本提示逻辑未完全失效，只是改成延迟/懒触发
5. `/healthz` 在页面打开期间的排队风险下降

### Step 4：只有在 Phase 1 完成后，才进入 Phase 3 / Phase 4 设计

你可以继续：

- 写 gevent 评估方案；或
- 写 scheduler 拆分设计草案；

但前提是先把首屏降载链路吃透并稳定。
