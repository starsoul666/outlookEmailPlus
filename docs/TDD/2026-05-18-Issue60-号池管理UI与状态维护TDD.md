## 1. 文档目标

本 TDD 只关注 Issue #60 的测试设计，不重复展开业务背景与技术方案。

本次测试应覆盖七类验证目标，其中第 5 项属于增强项路径：

1. 内部号池管理查询能正确区分**池内 / 池外**
2. 管理员可通过内部接口完成 **移入号池 / 移出号池**
3. `claimed` 占用中账号不会被通用动作误改
4. 基础状态维护动作遵守允许状态集合
5. 强制释放（若落地）只作用于 `claimed`，且不开放为任意强改
6. 新增内部管理接口不会污染现有 `GET /api/accounts` 与 external pool 语义
7. 新前端入口与交互契约与文档一致

术语约定：

1. **MVP**：只覆盖“池内/池外可见、移入、移出、`claimed` 保护”
2. **增强项**：强制释放、审计、最近流转展示
3. **通用动作**：`move_into_pool / move_out_of_pool / restore_available / freeze / retire`

## 3. 测试原则

1. **先锁状态机，再测 UI**：本需求最关键的是动作边界，不是页面长什么样
2. **MVP 与增强项分层断言**：不能把强制释放当作 MVP 必须项
3. **`claimed` 保护必须独立显式覆盖**：不能只靠其他失败用例顺便证明
4. **新接口与旧接口隔离验证**：要明确证明 `GET /api/accounts` 与 external pool 契约未被污染
5. **空结果也要合法**：池外为空、池内为空、最近流转为空时都不能 500
