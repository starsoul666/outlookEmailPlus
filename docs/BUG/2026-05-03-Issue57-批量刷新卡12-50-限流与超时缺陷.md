# 批量刷新 Outlook 账号：进度卡在 12/50（BUG）

**创建日期**: 2026-05-03  
**关联 Issue**: https://github.com/ZeroPointSix/outlookEmailPlus/issues/57  
**关联模块**: `outlook_web/services/graph.py`、`outlook_web/services/refresh.py`、`outlook_web/controllers/accounts.py`、`static/js/main.js`  
**状态**: 🟡 分析完成，等待报告者补充信息后决策修复方案  
**优先级建议**: P0（核心手动刷新链路，影响批量运营场景）

---

## 1. 问题概述

用户在前端全选 50 个 Outlook 邮箱后执行"刷新 Token"，SSE 进度每次都稳定卡在 **12/50**，第 13 个账号永远无法完成。用户猜测是 API 限流导致，建议增大间隔并增加失败重试。

版本：**v2.3.0**

---

## 2. 现场证据

### 2.1 用户描述特征

- 触发路径：全选 50 个账号 → 点击"刷新 Token"（即 `batchRefreshSelected`）
- 复现率：**100%**，每次必现
- 卡点位置：**永远卡在 12/50**，不是随机位置
- 表现：进度 Toast 停留，无报错，无自动恢复

### 2.2 数字"12"的隐含信息

每次固定卡在同一个位置（第 12 个成功，第 13 个 hang 住），强烈暗示存在**外部阈值**：

| 可能来源 | 阈值机制 | 表现特征 |
|---------|---------|---------|
| Microsoft OAuth token endpoint | 单 IP / 单应用注册限流 | 约 12 次请求后触发 429 或静默丢弃连接 |
| 代理工具（Clash/V2Ray 等） | 连接池上限或并发策略 | 第 13 条连接被阻塞或丢弃 |
| 防火墙/网关 | 高频连接检测 | 达到阈值后 RST 或静默丢包 |

> 本地测试无法复现，因环境和请求量差异，Microsoft 侧限流行为与生产环境不一致。

## 3. 深度根因分析

### 3.1 全链路梳理

```
[前端] batchRefreshSelected()
   ↓ POST /api/accounts/refresh/selected
[Controller] api_refresh_selected_accounts()
   ↓ yield from stream_refresh_selected_accounts()
[Service] stream_refresh_selected_accounts()
   ↓ loop: test_refresh_token_with_rotation()
[Graph] test_refresh_token_with_rotation()
   ↓ requests.post() → login.microsoftonline.com
[外部] Microsoft OAuth token endpoint
```

### 3.2 后端 Graph 层：`test_refresh_token_with_rotation()`

**位置**：`outlook_web/services/graph.py:263`

```python
def test_refresh_token_with_rotation(
    client_id: str,
    refresh_token: str,
    proxy_url: str = None,
    *,
    tenant: str = "common",
    scope: str = DEFAULT_GRAPH_SCOPE,
) -> tuple[bool, str | None, str | None]:
    try:
        proxies = build_proxies(proxy_url)
        resolved_scope = (scope or DEFAULT_GRAPH_SCOPE).strip() or DEFAULT_GRAPH_SCOPE
        res = requests.post(
            build_token_url(tenant),
            data={...},
            timeout=30,
            proxies=proxies,
        )
        ...
```

**缺陷**：

1. **零重试**：单次请求失败即返回失败，无自动重试
2. **未处理 429**：Microsoft 返回 429 (Too Many Requests) 时，未读取 `Retry-After` 响应头，也未执行退避等待
3. **timeout=30 过长**：在批量刷新场景下，如果第 13 个请求被静默丢弃（不发 RST），`requests` 可能 hang 满 30 秒才超时，期间整个 SSE 流停滞
4. **异常过于宽泛**：外层 `except Exception` 吞掉所有异常细节，返回模糊的"请求异常"

### 3.3 后端 Service 层：`stream_refresh_selected_accounts()`

**位置**：`outlook_web/services/refresh.py:670`

**循环逻辑**：

```python
for index, account in enumerate(accounts, 1):
    ...
    success, error_msg, new_refresh_token = test_refresh_token(client_id, refresh_token, proxy_url)
    ...
    if index < total and delay_seconds > 0:
        yield f"data: ...delay..."
        time.sleep(delay_seconds)
```

**缺陷**：

1. **串行阻塞**：每个账号的 `test_refresh_token` 是同步阻塞调用，如果某一次 hang 住，整个 generator 停止 yield，前端收不到任何 progress 事件
2. **delay 在请求之后**：`delay_seconds`（默认 5 秒）只在**成功请求后**执行，而不是在请求前。这意味着第 1~12 个请求之间几乎没有间隔（仅受网络延迟限制），极易触发限流
3. **无单账号超时保护**：未对 `test_refresh_token` 设置更短的超时（如 10 秒），也未在超时时中断并标记失败继续下一个

### 3.4 后端 Controller 层：SSE 响应头缺失

**位置**：`outlook_web/controllers/accounts.py:2324` / `2677`

```python
def generate():
    yield from refresh_service.stream_refresh_all_accounts(...)

return Response(generate(), mimetype="text/event-stream")
```

**缺陷**：

1. **缺少 `Cache-Control: no-cache`**：中间件或浏览器可能缓存 SSE 响应
2. **缺少 `Connection: keep-alive`**：长连接可能被提前关闭
3. **缺少 `X-Accel-Buffering: no`**：Nginx 等反向代理默认会缓冲响应，导致 progress 事件无法实时到达前端
4. **无显式 flush**：`yield` 后依赖 Python/WSGI 自动 flush，在部分部署环境下（如 gunicorn + gevent）可能存在缓冲延迟

### 3.5 前端层：`batchRefreshSelected()`

**位置**：`static/js/main.js:4100`

```javascript
async function batchRefreshSelected(accountIds) {
    const response = await fetch('/api/accounts/refresh/selected', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ account_ids: accountIds })
    });

    const reader = response.body.getReader();
    const decoder = new TextDecoder();
    let buffer = '';

    while (true) {
        const { done, value } = await reader.read();
        if (done) break;
        // 解析 SSE data: {...}
    }
}
```

**缺陷**：

1. **fetch 无超时**：`fetch` 默认无超时，如果后端 generator 停止 yield，前端会永远 `await reader.read()`
2. **无 AbortController**：无法主动中断卡住的请求
3. **无心跳检测**：SSE 流中无心跳事件，前端无法区分"后端处理慢"和"连接已死"
4. **break 条件不完整**：仅在 `data.type === 'complete' || data.type === 'error'` 时 break，如果后端既不发送 complete 也不发送 error（如 generator 挂起），循环永不退出

### 3.6 锁 TTL 计算逻辑

**位置**：`outlook_web/services/refresh.py:105`

```python
def compute_refresh_lock_ttl_seconds(total: int, delay_seconds: int) -> int:
    estimated = int(total * (max(delay_seconds, 0) + 2) + 600)
    ttl = max(REFRESH_LOCK_TTL_SECONDS, estimated)
    return min(ttl, 60 * 60 * 24)
```

其中 `REFRESH_LOCK_TTL_SECONDS = 60 * 60 * 2`（2 小时）。

以 50 个账号、delay=5 秒计算：

```
estimated = 50 * (5 + 2) + 600 = 950 秒
ttl = max(7200, 950) = 7200 秒 = 2 小时
```

这意味着一旦刷新任务因网络 hang 住而无法正常结束，锁会持有 2 小时，期间用户无法发起新的刷新任务（会报 `REFRESH_CONFLICT`）。

## 4. 影响范围

### 4.1 直接影响

1. **批量刷新 selected**（`/api/accounts/refresh/selected`）：用户报告的核心路径
2. **全量刷新**（`/api/accounts/refresh-all`）：同样使用 `stream_refresh_all_accounts()`，若账号数量 > 12，同样会触发
3. **定时刷新**（`/api/accounts/trigger-scheduled-refresh`）：scheduled 路径同样受影响

### 4.2 间接影响

1. **刷新锁长时间占用**：若任务异常中断，锁 TTL 2 小时，期间所有刷新操作被阻塞
2. **用户体验受损**：进度永远卡住，无报错，用户无法判断是网络问题还是系统故障
3. **运营效率下降**：批量运营场景（如 50/100 个账号）下无法完成 token 维护

### 4.3 不影响

1. **单账号刷新**（`/api/accounts/<id>/refresh`）：单个请求不会触发限流阈值
2. **IMAP 账号**：已被过滤排除，不进入 OAuth 刷新链路
3. **CF Worker 临时邮箱**：同样被过滤排除

## 5. 修复方案建议（分阶段）

### Phase 1：后端重试与 429 退避（最高优先级）

**目标**：缓解 Microsoft OAuth 限流导致的 hang 住问题

**修改点**：`outlook_web/services/graph.py`

1. `test_refresh_token_with_rotation()` 增加 `max_retries` 参数（默认 3 次）
2. 遇到 429 时读取 `Retry-After` 头并按其值等待；若无则指数退避（1s / 2s / 4s）
3. 遇到 5xx 或网络异常时同样重试
4. 400/401 等明确错误不再重试，直接返回失败
5. 将 timeout 从 30 秒缩短至 15 秒（批量场景下更合理）

### Phase 2：SSE 响应头与 flush 优化（高优先级）

**目标**：确保 progress 事件实时到达前端，不被中间件缓冲

**修改点**：`outlook_web/controllers/accounts.py`

1. 所有 SSE Response 统一添加：
   - `Cache-Control: no-cache`
   - `Connection: keep-alive`
   - `X-Accel-Buffering: no`
2. 考虑在 generator 中每次 yield 后显式 flush（若 WSGI 环境支持）

### Phase 3：前端 SSE 超时与中断保护（高优先级）

**目标**：防止前端永远等待卡住的 SSE 流

**修改点**：`static/js/main.js`

1. `fetch` 使用 `AbortController` 设置整体超时（如 120 秒）
2. 增加心跳检测：若超过 30 秒未收到任何 SSE 事件，主动中断并提示用户
3. 优化 break 条件：在超时或连接异常时也能退出循环

### Phase 4：刷新间隔策略优化（中优先级）

**目标**：降低触发限流的概率

**修改点**：`outlook_web/services/refresh.py`

1. 将 `delay_seconds` 的生效时机从"请求后"改为"请求前"，确保每次请求前有固定间隔
2. 或在 `test_refresh_token_with_rotation()` 调用前增加随机 jitter（如 0~2 秒），避免请求过于规整触发限流
3. 评估增大默认 `refresh_delay_seconds`（从 5 秒调整为 8~10 秒）
